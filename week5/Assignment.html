Work Products Due Next Week

These documents should be avaible in your CM system. Please give read/write access to me and Sarun (mapfaps@gmail.com, sarun.w@ku.th).

  1. Software Development Plan with Schedule, Effort, and PV estimates. All sections of the SDP should be written.

  2. Schedule in SDP should have a table of iterations showing what features or UC will be done in each iteration, and what work products will be produced. The Schedule should show when you will implement all the UC and other requirements. You should schedule approx. the final 10% of time for system testing and fixing problems.

    The PMC and PPQA should verify the schedule against the SRS. Are all requirements included? Are the time estimates consistent and reasonable?

  3. Coding Standard (Developers): Create a coding standard for your source code. It should conform to a commonly used standard and include rules for the following:
    1. code format;
    2. names of variables, methods, classes, packages, and files. Please don't use Hungarian notation or prefixing (such as "m" for "members") unless that's really the language standard;
    3. in-code documentation and comments;
    4. directory structure, including where to put source code and test code;
    5. package structure,
    6. anything else that is relevant to source code!
    The coding standard should include an example of all the rules. You should date it, so that if there are revisions later then people will know whether they have the current version.

    Tooling Support: Look for programming aids to make it easier to use your coding standard, and to verify code adheres to standard. In Eclipse, language-specific formatting is in Preferences -> Java -> Code Style -> Formatting (for Java); you can create and import your own formatting rules. To check your code in Eclipse, enable Checkstyle for a project and supply your own Checkstyle rules for your coding standard (there are many checkstyle rule sets on the Internet).

  4. Test Plan (Testers): Create a form and instructions for planning and performing tests. There are lots of test forms on the Internet; study a few and choose what is suitable for your project. A test plan helps you design tests and ensures you don't forget anything during retesting. The form is used to report actual versus expected results, and identifies the work being tested (including version). A test plan can be used for both manual and automatic testing, e.g. xUnit.

  5. Baseline Identification & Management (CM): Clearly document in your CM PLan how will you identify and manage baselines. Educate your team on what is a baseline and the standard for baselines.

    A baseline provides a fixed reference. You use a baseline schedule to compare actual progress and effort against your plan. A baseline SRS provides requirements you can design from, and if requirements change, a baseline is needed to assess the change. A software baseline identifies work that has achieved some milestone or has been tested for some behavior.

    Your CM plan for baselines should specify:

    • What products are baselined?
    • When will you create a baseline?
    • How is a baseline identified?
    • After a baseline is created, the "baselined" items should not be modified.
    • You should specify baselines for important docs like SRS and SDP. The Schedule in SDP should be preserved so original schedule can be compared to actual schedule, for learning.
    • Baseline source code at important milestones, deliverables such as end of iteration product, and releases. Unlike documents, you can use the VCS capabilities to identify baselines.

  6. Update to Weekly WAR (PMC):

    • (Optional) EV of tasks done this week. EV is the PV (plan value) from the SDP.
    • You should check that the report is accurate.

Useful Books

Jalote, Software Project Managment in Practice
This short book describes project management at Infosys. I think it is a useful guide for some of your roles. It has a chapters on Project Planning, Configuration Management, PMC, Quality Planning, the Project Management Plan, and Reviews. You can read the first few chapters at Google Books.
Stellman & Greene, Applied Software Project Managment
Easy to understand descriptions of many of the practices you are using, including script-like guidance for some activities.
