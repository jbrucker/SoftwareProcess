These documents should be avaible in your CM system. Please give read/write access to me and Sarun (mapfaps@gmail.com, sarun.w@ku.th).

  1. Coding Standard (Developers): Create a coding standard for your source code. It should conform to a commonly used standard and include rules for the following: (a) code format; (b) names of variables, methods, classes, packages, and files. Please don't use Hungarian notation or prefixing (such as "m" for "members") unless that's the language standard; (c) in-code documentation and comments; (d) directory structure, including where to put source code and test code; (e) package structure, (f) anything else that is relevant to source code!

    The coding standard should include an example of all the rules. You should date it, so that if there are revisions later then people will know whether they have the current version.

  2. Test Plan (Testers): Create a form and instructions for planning and performing tests. There are lots of test forms on the Internet; study a few and choose what is suitable for your project. A test plan helps you design tests and ensures you don't forget anything during retesting. The form is used to report actual versus expected results, and identifies the work being tested (including version). A test plan can be used for both manual and automatic testing, e.g. xUnit.
  3. Baseline Identification & Management (CM): Clearly document in your CM PLan how will you identify and manage baselines. Educate your team on the standard for baselines. This includes:
    • What products are baselined?
    • When will you create a baseline?
    • How is a baseline identified?
    • After a baseline is created, the "baselined" items should not be modified.
    • You should specify baselines for important docs like SRS and SDP. The Schedule in SDP should be preserved so original schedule can be compared to actual schedule, for learning.
    • You should baseline source code at important milestones, deliverables such as end of iteration product, and other releases. Unlike documents, you can use the VCS capabilities to identify baselines.
  4. Plan Value and Earn Value Monitoring (PMC): How do you know if you are on-track for this iteration? Assign earned value to all product-related tasks in the SDP and iteration plan.

    Useful Books

    Jalote, Software Project Managment in Practice
    This short book describes project management at Infosys. I think it is a useful guide for some of your roles. It has a chapters on Project Planning, Configuration Management, PMC, Quality Planning, the Project Management Plan, and Reviews. You can read the first few chapters at Google Books.
    Stellman & Greene, Applied Software Project Managment
    Easy to understand descriptions of many of the practices you are using, including script-like guidance for some activities.
